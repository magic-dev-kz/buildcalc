### 8. escapeHtml -- does it escape quotes?
**PASS (improved).** The `escapeHtml()` function (lines 619-623) uses `document.createElement('div') + createTextNode + innerHTML` approach. This escapes `<`, `>`, `&` automatically.

For the specific concern about attribute injection: looking at line 1079:
```
data-id="'+escapeHtml(p.id)+'"
```
The `p.id` is generated by `Date.now().toString(36)+Math.random().toString(36).slice(2,6)` (line 1049), so it's always alphanumeric. However, `escapeHtml()` via `createTextNode().innerHTML` does NOT escape `"` or `'`. If `p.id` were user-controlled and contained `"`, it could break out of the attribute.

**BUG-S01 (P3):** `escapeHtml()` does not escape `"` or `'`. Currently safe because `p.id` is auto-generated, but `p.name` IS user-controlled and is used in `aria-label="Delete project '+escapeHtml(p.name||'Untitled')+'"` (line 1079). A project name containing `"` would break the attribute boundary. The `createTextNode + innerHTML` technique only escapes `<`, `>`, `&`, NOT quotes.

**Repro:** Save a project with name `test" onclick="alert(1)`, then open projects panel. The `"` in name will break out of the aria-label attribute. This is a self-XSS (user injecting themselves), but still incorrect behavior.

## Edge Case Testing -- "Try to BREAK it"

### Zero / Empty Values
**PASS.** `requirePositive()` rejects null/0/negative. Toast message shown. Error styling applied. Focus moved to first error field.

### Negative Values
**PASS.** `getNum()` returns null for negative (line 649: `v<0` check). `min="0"` on HTML inputs provides browser-level prevention too.

### Huge Numbers
**BUG-E01 (P3):** Entering `999999999` for all dimensions in concrete calculator:
- Volume = ~999999999^3 meters cubed = astronomically large number
- JavaScript handles it (no crash), but displays as scientific notation-like huge numbers
- Bags needed = enormous number, result text overflows visually
- No practical maximum validation exists. Not a crash but produces meaningless results.

### L-Shape Cutout Validation
**PASS.** Lines 828: `if(cw>=tw||ch>=th)` correctly rejects cutouts larger than total. Good.

### Paint Calculator -- Zero Doors/Windows
**PASS.** Defaults to 0 if null (lines 882-883). Works correctly.

### Flooring Custom Pack -- Zero
**PASS.** `requirePositive` catches it.

### Unit Converter -- Negative Input
**BUG-E02 (P2):** Converter `conv-from` and `conv-to` inputs have no `min="0"` and `getNum` is not used -- raw `parseFloat` is used (lines 1017, 1024). Entering negative values converts them (e.g., -10 feet = -3.048 meters). This is mathematically correct but physically meaningless for construction measurements. Minor issue since negative lengths don't exist in construction.

### Tile Waste = 0
**PASS.** Works correctly, just calculates without waste.

### Tile Waste = 100 or higher
**BUG-E03 (P3):** `tile-waste` has `max="50"` in HTML but no JS validation for values > 50. Browser native validation only works on form submit, not on button click. User can type `200` and get 200% waste calculation. Same for `floor-extra-waste`.

## Code Quality Issues

**BUG-Q01 (P3):** `showResult` uses `innerHTML` for the detail parameter (line 694: `$(id+'-detail').innerHTML=detail||''`). All callers do escape via `escapeHtml()`, so it's safe currently. But the pattern is fragile -- any future caller forgetting to escape would create XSS.

**BUG-Q02 (P3):** `renderProjects` uses `el.innerHTML = projects.map(...)` (line 1072). Project name IS escaped, but the overall innerHTML pattern is risky for maintainability.

**OBSERVATION:** No inline `onclick` handlers remain. All event listeners attached via `addEventListener`. This was an issue in v1 -- FIXED.
